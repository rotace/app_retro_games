# 2. workspace-structure

Date: 2026-06-12

## Status

Accepted

## Context

プロジェクト憲法に基づき、本ゲームシステムは「コア層（純粋なゲームロジック）」と「ランナー層（Debianの実機環境、およびWSLgのシミュレーション環境）」の2層データ駆動構造を採用する。
コア層の独立性を保ち、ランナー層への逆依存や外部の画面描画ライブラリがコア層に混入するのをビルドレベルで防ぐため、Rustの **Cargo Workspace** 機能を活用したディレクトリ構造を厳格に定義する必要がある。

## Decision

プロジェクトのルートに Cargo Workspace を導入し、以下のように各パッケージ（クレート）を物理的に分離して配置する。

### 提案するディレクトリ構造

```text
.
├── Cargo.toml        # Workspace全体の管理
│
├── core/             # 【コア層】
│   ├── Cargo.toml    # 外部描画ライブラリへの依存禁止、no_std対応可能
│   └── src/
│       └── lib.rs    # ランナー層との結合境界となるエントリ（関数・データ構造）を定義
│
├── runner-debian/    # 【ランナー層：実機環境】
│   ├── Cargo.toml    # core に依存。Linux特有のクレートを使用
│   └── src/
│       └── main.rs   # 実機環境の初期化、コア層のエントリ呼び出し、メインループの制御
│
└── runner-wslg/      # 【ランナー層：シミュレーション環境】
    ├── Cargo.toml    # core に依存。ウィンドウ描画関連のクレートを使用
    └── src/
        └── main.rs   # 仮想環境の初期化、コア層のエントリ呼び出し、メインループの制御

```

### 依存関係のルール

- `runner-debian` と `runner-wslg` は、`core` パッケージをローカルパス依存（`path = "../core"`）として取り込む。
- `core` は、いかなる `runner-*` パッケージ、およびUI/ウィンドウ描画関連の外部クレートにも依存してはならない。
- 各クレートの内部設計（`core` 内のモジュール分割や詳細なファイル構成）は本ADRの対象外とし、今後の実装フェーズにおいて原則に沿って決定する。

## Consequences

### Positive

- `core` の `Cargo.toml` を確認するだけで、外部環境（OS/GUI）に依存していないことが一目で担保できる。
- ルート直下に各クレートが並ぶRustの標準的なWorkspace構成となるため、開発時のパス指定やビルドコマンドが直感的になる。
- 各ランナー層が独立したバイナリ（`main.rs`）を持つため、ビルドターゲット（クロスコンパイル等）の切り分けが容易になる。


### Negative

- パッケージが分かれるため、結合境界で利用する型定義（入力状態やVRAM抽象化など）を必ず `core` 側に定義させる運用の規律が必要となる。（ただし、これは憲法の設計思想通りであるため許容する）
