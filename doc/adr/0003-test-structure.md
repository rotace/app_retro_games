# 3. test-structure

Date: 2026-06-12

## Status

Accepted

## Context

Rust ではテストを以下の 2 種類に大別できる

- **単体テスト（Unit Test）**: モジュール内部の関数や構造体の振る舞いを、依存を最小化して検証するテスト。
- **結合テスト（Integration Test）**: crate 全体の公開 API を通じて、複数モジュールの連携を検証するテスト。

Rust の標準的な構造では
- **単体テスト** → 各ファイル内の mod tests
- **結合テスト** → tests/ ディレクトリ
という慣習がある。

しかし AI コーディング（spec-kit）では、テストの配置方針を明確にしないと
- LLM が tests/ に単体テストを生成してしまう
- モジュール内テストと結合テストが混在し、責務が曖昧になる
- テストの粒度が揺れ、レビューコストが増大する
- 仕様変更時に AI が誤った場所にテストを再生成する

という問題が発生する。

そのため、テスト配置のルールを ADR として明文化し、AI にも一貫した指示を与える必要がある。

## Decision

Rust プロジェクトにおいて、以下のテスト配置方針を採用する。

### 単体テスト（Unit Test）

- 各実装ファイルの末尾に mod tests として配置する
- テスト対象のモジュールと同一ファイルに置く
- 非公開関数・内部構造のテストを許可する（super::* を使用）

### 結合テスト（Integration Test）

- tests/ ディレクトリに配置する
- crate の公開 API（public interface）を通じて動作を検証する
- 複数モジュールの連携や CLI の end-to-end 動作を対象とする

### AI コーディング時のルール

- `/speckit.plan`実行時に、必要に応じて本ADRを引用する。
- implement 実行時に LLM が tests/ に単体テストを生成しないようにする
- 結合テストは tests/ に生成するよう明確に指示する

## Consequences

### Positive

- テストの粒度が明確になり、レビューが容易になる
- AI が一貫した場所にテストを生成するようになる
- private 関数のテストが容易になり、品質が向上
- 結合テストが crate の公開 API に集中するため設計が洗練される

### Negative

- 単体テストが増えるとファイル末尾が長くなる
- 結合テストで内部構造を直接テストできない
- モジュール分割が不適切だと単体テストが複雑化する
